## 1. 系统概述

### 1.1 定位

基于 CET-4 大纲词汇表（4,517词）的个人自适应背单词系统。支持多用户注册，每人独立进度，局域网内访问。

### 1.2 核心能力

| 能力 | 说明 |
|------|------|
| **间隔重复** | 基于 SM-2 算法动态安排复习计划 |
| **自适应混入** | 复习→测试全通过后，才混入本轮未学新词 |
| **多维熟练度** | 综合正确率、反应时间、遗忘曲线等多维度评估 |
| **多轮次** | 全部词汇掌握率达 90% 即完成一轮，可进入下一轮 |
| **混合答题** | 选择题快速过 + 拼写输入加深记忆 |
| **可配置** | 每场次新词数、每日提醒等用户可调 |
| **时间预算** | 每次学习控制在 30~40 分钟，默认参数按此校准 |

### 1.3 用户画像

- 备考 CET-4 的大学生
- 需要系统化、可量化的背单词工具
- 每天固定时间学习，偏好有节奏的闯关模式

## 4. SM-2 间隔重复算法

### 4.1 算法原理

SM-2 (SuperMemo 2) 是 1987 年由 Piotr Woźniak 提出的间隔重复算法，至今仍是 Anki 等主流软件的核心算法。

核心思想：**每次复习后，根据记忆质量动态计算下次复习时间。**

### 4.2 关键变量

| 变量 | 含义 | 初始值 | 范围 |
|------|------|--------|------|
| `EF` | 易度因子 (Easiness Factor) | 2.5 | [1.3, 3.0] |
| `I(n)` | 第 n 次复习后的间隔(天) | — | ≥1 |
| `q` | 本次答题质量评分 | — | [0, 5] |
| `reps` | 连续正确次数 | 0 | ≥0 |

### 4.3 自动质量评分 (q)

```typescript
function calculateQuality(correct: boolean, responseTimeMs: number): number {
  if (!correct) {
    return 0;  // 错误 → 0分
  }

  // 正确，按反应时间评分
  if (responseTimeMs < 2000)  return 5;  // 极快，几乎自动反应
  if (responseTimeMs < 5000)  return 4;  // 正常速度
  if (responseTimeMs < 10000) return 3;  // 较慢，需要思考
  if (responseTimeMs < 20000) return 2;  // 很慢，犹豫很久
  return 1;                               // 超慢但正确
}
```

> **说明**：SM-2 原版要求用户自评（0-5），本系统根据客观数据自动计算，消除主观偏差。
> 选择题和拼写题使用不同的时间阈值（选择题更快，拼写题适当放宽）。

### 4.4 核心公式

```typescript
interface CardState {
  ef: number;           // Easiness Factor
  interval: number;     // 当前间隔（天）
  repetitions: number;  // 连续正确次数
}

function sm2_step(state: CardState, quality: number): CardState {
  let { ef, interval, repetitions } = state;

  if (quality < 3) {
    // 失败：重置连续计数，但保留 EF
    repetitions = 0;
    interval = 1;
  } else {
    // 成功：更新 EF 和间隔
    // EF 更新公式
    ef = ef + (0.1 - (5 - quality) * (0.08 + (5 - quality) * 0.02));
    // EF 限制在 [1.3, 3.0]
    ef = Math.max(1.3, Math.min(3.0, ef));

    // 间隔计算
    repetitions += 1;
    if (repetitions === 1) {
      interval = 1;       // 第一次 → 1天
    } else if (repetitions === 2) {
      interval = 6;       // 第二次 → 6天
    } else {
      interval = Math.round(interval * ef);  // 之后每次 × EF
    }
  }

  return { ef, interval, repetitions };
}
```

### 4.5 失败处理

- **当次处理**：答错的词在当前场次内标记，立即进入 retest 队列
- **retest 规则**：同一场次内反复测试错词，直到答对（q≥3）
- **跨场次**：下次开启新场次时，上次错词优先出现在复习列表
- **EF 不重置**：失败只重置 reps 和 interval，保留 EF（SM-2 标准做法）

### 4.6 学习中的特殊情况

| 情况 | 处理 |
|------|------|
| 选择题猜对但很慢 | q=2，视为"未真正掌握"，缩短下次间隔 |
| 拼写稍错（漏字母） | 视为错误，重新学习 |
| 连续3次同一词错误 | 降低 EF，增加出现频率 |
| 长期未复习的词 | 按当前 EF 计算已过期天数，加权紧急度 |

## 5. 熟练度评估模型

### 5.1 评估维度及权重

| 维度 | 权重 | 说明 | 计算方式 |
|------|------|------|----------|
| **正确率** | 35% | 历史正确比例 | `total_correct / total_attempts` |
| **反应速度** | 25% | 平均响应时间越短越熟练 | 归一化: `1 - (avg_rt - min_rt) / (max_rt - min_rt)` |
| **连续正确** | 15% | 当前连续正确次数 | `min(consecutive / 5, 1.0)` 阈值5次 |
| **间隔稳定性** | 15% | 间隔是否稳定增长 | 最近3次间隔是否呈增长趋势 |
| **回忆努力度** | 10% | 首次尝试正确率 | 本词首次出现时是否一次答对 |

### 5.2 综合公式

```typescript
function calculateProficiency(wordState: UserWord, reviewHistory: ReviewLog[]): number {
  const accuracy = wordState.total_attempts > 0
    ? wordState.total_correct / wordState.total_attempts
    : 0;

  const speedScore = calculateSpeedScore(wordState.avg_response_time);

  const consistency = Math.min(wordState.consecutive_correct / 5, 1.0);

  const stability = calculateStability(reviewHistory);

  const firstAttempt = calculateFirstAttemptRate(reviewHistory);

  const proficiency = Math.round(
    accuracy * 0.35 +
    speedScore * 0.25 +
    consistency * 0.15 +
    stability * 0.15 +
    firstAttempt * 0.10
  );

  return Math.max(0, Math.min(100, proficiency));
}
```

### 5.3 熟练度等级

| 等级 | 分值 | 标签 | 行为 |
|------|------|------|------|
| Lv0 | 0 | 未学习 | 从未见过 |
| Lv1 | 1-25 | 初识 | 刚学过，还记不住 |
| Lv2 | 26-50 | 学习中 | 需要提示才能回忆 |
| Lv3 | 51-75 | 基本掌握 | 能正确回答但较慢 |
| Lv4 | 76-89 | 较熟练 | 快速正确回答 |
| **Lv5** | **90-100** | **已掌握** | 计入轮次进度，间隔大幅拉长 |

### 5.4 遗忘风险

```typescript
// 对于已掌握(Lv5)的词，计算遗忘风险
function calculateForgettingRisk(wordState: UserWord): number {
  const daysSinceReview = daysBetween(new Date(), wordState.last_reviewed_at);
  const overdue = daysSinceReview - wordState.interval_days;

  if (overdue <= 0) return 0;  // 还没到复习时间

  // 超过间隔越久，遗忘风险越大
  const risk = Math.min(overdue / wordState.interval_days, 1.0);
  return risk;
}
```

### 5.5 与 Ebbinghaus 遗忘曲线的关系

SM-2 算法的间隔增长本质上是 Ebbinghaus 遗忘曲线的逆向应用：

```
记忆保持率
  100% ┤
       │    ┌──┐        ┌──┐        ┌──┐        ┌──┐
   90% ┤    │  │        │  │        │  │        │  │
       │    │  │        │  │        │  │        │  │
   75% ┤    │  │   ←复习→  │   ←复习→  │   ←复习→  │
       │    │  │        │  │        │  │        │  │
   50% ┤    │  │        │  │        │  │        │  │
       │    │  │        │  │        │  │        │  │
   25% ┤    │  │        │  │        │  │        │  │
       │    │  │        │  │        │  │        │  │
    0% ┼────┴──┴────────┴──┴────────┴──┴────────┴──┴──▶ 时间
           1天         6天        6*EF天     6*EF²天
```

每次成功复习"重置"遗忘曲线，且下次遗忘速度变慢（曲线更平缓）。

| 概念 | 英文 | 定义 |
|------|------|------|
| **学习场次 / 场次** | Session | 一次连续的学习活动，完整走完「复习 → 学习新词 → 总测试」三个阶段。场次内答错的词进入重考循环。SQL 层对应 `sessions` 表。 |
| **轮 / 轮次** | Round | 由多个学习场次组成的完整循环。当本轮已学词汇中 90% 达到熟练度 ≥ 90 时轮次结束，进入下一轮。 |
| **阶段** | Phase | 一场学习内部的三个子过程：复习阶段 → 新词阶段 → 总测试阶段，顺序固定不可逆。 |
| **熟练度** | Proficiency | 0–100 的数值。基于 SM-2 算法的多维度评估（正确率、反应时间、连续正确数等）综合计算。 |
| **已掌握** | Mastered | 熟练度 ≥ 90 的词，视为已掌握。掌握率达 90% 即算一轮通关。 |
| **重考** | Retest | 总测试阶段答错的词进入重考队列，循环测试直到全部答对，才能完成当前场次。 |
| **学习模式** | New-word Mode | 每轮开始时从 28 个选项中选择一个：「全随机」从全部词库随机抽取、「按首字母乱序」按 A→Z 分组推进、或「仅学某字母开头的单词」（A~Z），本轮内不可更改。 |

### 6.2 复习阶段的排序算法

```typescript
function getReviewQueue(userWords: UserWord[]): UserWord[] {
  return userWords.sort((a, b) => {
    // 1. 先排上次场次中标记为"困难"的词
    if (a.was_difficult !== b.was_difficult) {
      return a.was_difficult ? -1 : 1;
    }

    // 2. 按遗忘风险降序（最危险的最先复习）
    const riskA = calculateForgettingRisk(a);
    const riskB = calculateForgettingRisk(b);
    if (Math.abs(riskA - riskB) > 0.1) {
      return riskB - riskA;
    }

    // 3. 按熟练度升序（最不熟的先复习）
    return a.proficiency - b.proficiency;
  });
}
```

### 6.3 新词混入策略

每轮开始时由用户选择一种模式，本轮内固定不变。

#### 模式 A：全随机（默认）

全词汇表内随机抽取未学词，不依赖任何顺序。

```typescript
function getNewWords_random(userId: number, round: number, count: number): Word[] {
  return db.query(`
    SELECT w.* FROM words w
    WHERE w.id NOT IN (
      SELECT uw.word_id FROM user_words uw
      WHERE uw.user_id = ? AND uw.round = ?
    )
    ORDER BY RANDOM()
    LIMIT ?
  `, [userId, round, count]);
}
```

#### 模式 B：按首字母乱序

按首字母 A→Z 分组推进，组内随机。例如先学完所有 A 开头的词，再学 B...

```typescript
function getNewWords_alpha(userId: number, round: number, count: number): Word[] {
  // 先找出当前用户本轮已学到哪个首字母
  const lastLearned = db.query(`
    SELECT substr(w.word, 1, 1) AS initial
    FROM user_words uw
    JOIN words w ON w.id = uw.word_id
    WHERE uw.user_id = ? AND uw.round = ?
    ORDER BY initial DESC
    LIMIT 1
  `, [userId, round]);

  const currentInitial = lastLearned.length > 0
    ? lastLearned[0].initial
    : 'A';  // 从头开始

  return db.query(`
    SELECT w.* FROM words w
    WHERE w.id NOT IN (
      SELECT uw.word_id FROM user_words uw
      WHERE uw.user_id = ? AND uw.round = ?
    )
    ORDER BY
      CASE
        WHEN substr(w.word, 1, 1) < ? THEN 1   -- 已过字母排最后
        WHEN substr(w.word, 1, 1) = ? THEN 2    -- 当前字母优先
        ELSE 3
      END,
      substr(w.word, 1, 1),    -- 同字母优先
      RANDOM()                 -- 组内随机
    LIMIT ?
  `, [userId, round, currentInitial, currentInitial, count]);
}
```
  
#### 模式 C：仅学某字母开头（A~Z）

限定本轮只学某首字母的单词。与模式 A/B 平级选择，28 个选项互斥：

```
全部词汇：
  · 全随机          — 所有 4517 词，每次随机抽取
  · 按首字母乱序     — 所有词，A→Z 分组推进

仅学某字母开头：
  · A B C D E F G … Z — 仅该字母开头的单词（选字母后相当于随机抽取该字母词库）
```

影响范围：
- `/learn/new-words` — 抽取新词时加 `WHERE upper(w.word) LIKE 'E%'`（非全词模式时）
- `/learn/review-queue` — 复习队列中同样按 scope 过滤
- `user_settings.new_word_mode` 存储当前轮的选择：`'random'` / `'alpha'` / 单字母 `'A'`~`'Z'`（复用现有字段，无需新增列）

> ⚠️ **选择题干扰项不受 scope 限制**，始终从全局词库随机抽取（`GET /learn/distractors` 不做 scope 过滤），避免选项同首字母导致线索泄露。

### 6.4 答题模式

#### 模式A: 选择题 (四选一)

```
Q: 请选择 "abandon" 的中文意思

  A. 接受          B. 抛弃 ✓
  C. 关于          D. 抽象

用户选择 → 即时反馈正误 → 记录耗时
```

- 干扰项优先从全量字典随机选取（页面初始化时预取 50 条），不足时从当前学习队列补足
- 选项位置随机排列

#### 模式B: 拼写题

```
Q: 请拼写 "丢弃；放弃，抛弃"

输入: [________]

提示: a _ _ _ _ _ _ (首字母提示)
```

- 拼写题有首字母提示（可配置开关）
- 大小写敏感
- 拼写错误率高的词会增加拼写题出现频率

#### 模式切换策略

由前端控制：

```
关闭拼写模式        → 纯选择题
开启拼写 + 前 2 题   → 选择题
开启拼写 + 第 3 题起 → 选择题/拼写题各 50% 随机
```

### 6.5 时间预算（30~40 分钟校准）

每次学习控制在 30~40 分钟，默认参数按此校准：

| 阶段 | 词数 | 单题耗时 | 小计 |
|------|------|----------|------|
| 复习到期词 + 错词 | ~20 词 | ~12 秒 | ~4 分钟 |
| 学习新词（预览 + 首次测试） | N = 15 | ~30 秒 | ~7.5 分钟 |
| 总测试（本轮已学全部词） | ~30 词 | ~12 秒 | ~6 分钟 |
| 错题重考 + 缓冲 | 浮动 | — | ~3 分钟 |
| **合计** | | | **~20-25 分钟（严格学习）** |

> **实际耗时通常更长**：思考停顿、中途被打断、查看统计、拼写题打字慢等因素。按经验，默认 N=15 时多数用户实际用时在 30~40 分钟之间。
>
> 用户可在设置中调高/调低「每轮新词数 N」来缩短或延长单次学习时长。

## 9. 多轮次系统

### 9.1 轮次逻辑

```
第1轮开始 ──▶ 学习全部4517词 ──▶ 熟练度≥90% ──▶ 第1轮完成
                                                    │
                                                    ▼
第2轮开始 ──▶ 重新学习全部词  ──▶ 熟练度≥90% ──▶ 第2轮完成
               ↑ 保留历史数据      ↑ 更快达到          │
               ↑ EF不重置          ↑ 因为基础更好       ▼
                                                    ...
```

### 9.2 轮次切换规则

| 条件 | 动作 |
|------|------|
| `proficiency >= 90` 的词数 >= `total * 90%` | 弹出"轮次完成"祝贺，进入下一轮 |
| 进入新轮 | 重置 `interval`, `repetitions`, 但保留 `ef`, `total_correct`, `total_attempts` |
| 新轮初始熟练度 | 按历史数据重新计算（比第一轮起点高） |

### 9.3 轮次初始化

```sql
-- 进入新轮时：创建新的 user_words 记录（保留历史数据）
INSERT INTO user_words (user_id, word_id, ef, round, total_correct, total_attempts, ...)
SELECT user_id, word_id, ef, round + 1, 0, total_correct, total_attempts, ...
FROM user_words
WHERE user_id = ? AND round = ?;

-- 或更简单：将 round 递增，重置 interval
UPDATE user_words
SET round = round + 1,
    interval_days = 0,
    repetitions = 0,
    -- 保留 ef, total_correct, total_attempts
    next_review = datetime('now')
WHERE user_id = ? AND round = ?;
```

#### start.sh（多模式脚本）

目前支持 6 种模式：

| 命令 | 说明 |
|------|------|
| `bash start.sh` | 构建 Docker 镜像并启动服务 |
| `bash start.sh --test` / `-t` / `test` | 测试模式，使用 50 词小数据库（含测试账号） |
| `bash start.sh build` | 在容器内构建前端（修改源码后刷新用，无需宿主机 Node.js） |
| `bash start.sh reset-password <用户名> [新密码]` | 重置用户密码（不指定新密码则自动生成随机密码） |
| `bash start.sh fix-ownership` | 修复 data/config/logs 所有权（容器创建的 root 文件） |
| `bash start.sh --help` / `-h` / `help` | 查看帮助 |

```bash
#!/usr/bin/env bash
set -e

PROJECT_DIR="$(cd "$(dirname "$0")" && pwd)"
cd "$PROJECT_DIR"

MODE="${1:-start}"  # start | --test | build | reset-password | fix-ownership | --help
USERNAME="${2:-}"
NEW_PASS="${3:-}"

# 当前用户 UID/GID（防止 Docker 创建的文件归属 root）
CURRENT_UID="$(id -u)"
CURRENT_GID="$(id -g)"

# ─── 帮助 ─────────────────────────────────────────
if [ "$MODE" = "--help" ] || [ "$MODE" = "-h" ] || [ "$MODE" = "help" ]; then
  echo "用法: bash start.sh [--test|-t] | build | reset-password <用户名> | fix-ownership"
  echo "  (默认)   构建并启动服务"
  echo "  --test   测试模式（50 词小数据库）"
  echo "  build    在容器内构建前端（改源码后刷新用）"
  echo "  reset-password  重置用户密码"
  echo "  fix-ownership   修复 data/config/logs 所有权"
  exit 0
fi

# ─── 启动模式 ────────────────────────────────────
if [ "$MODE" = "start" ]; then
  mkdir -p ./data ./config ./logs
  docker compose -f docker/docker-compose.yml up -d --build
  # 修复容器创建的文件所有权（防止 root 归属）
  sudo chown -R "$CURRENT_UID:$CURRENT_GID" ./data ./config ./logs 2>/dev/null || true
  ...
fi

# ─── 测试模式 ────────────────────────────────────
if [ "$MODE" = "--test" ] || [ "$MODE" = "-t" ] || [ "$MODE" = "test" ]; then
  docker compose -f docker/docker-compose.yml -f docker/docker-compose.test.yml up -d --build
  docker exec --user "$CURRENT_UID:$CURRENT_GID" cet4-web node cli/seed-test.js
  # 修复容器创建的文件所有权（防止 root 归属）
  sudo chown -R "$CURRENT_UID:$CURRENT_GID" ./data ./config ./logs 2>/dev/null || true
  # 创建 50 词 + test/test123 管理员账号
  ...
fi

# ─── 前端构建 ────────────────────────────────────
if [ "$MODE" = "build" ]; then
  docker run --rm --user "$CURRENT_UID:$CURRENT_GID" -v "$(pwd):/workspace" -w /workspace cet4-web:latest npm run build:client
  ...
fi
```

> **文件所有权防护**：脚本在以下环节自动修复 Docker 容器创建的文件归属问题：
> 1. `docker run`/`exec` 均使用 `--user $(id -u):$(id -g)` 运行，以宿主机用户身份操作
> 2. `docker compose up` 后执行 `sudo chown`，兜底修复
> 3. 独立命令 `bash start.sh fix-ownership` 提供手动修复入口
>
> **容器内构建前端**：即使宿主机没有 Node.js，也可以通过 `docker run --rm -v $(pwd):/workspace -w /workspace cet4-web:latest npm run build:client` 在容器内完成前端构建。输出写入宿主 `dist/client/`，浏览器刷新即可看到效果。
