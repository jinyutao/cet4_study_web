## 1. 日志引擎选型

### 1.1 选型对比

| 特性 | pino | winston | console |
|------|------|---------|---------|
| 运行时开销 | **最低**（~5x 快于 winston） | 较高 | 最低 |
| JSON 结构化 | 原生输出 JSON | 需配置 formatter | 手动拼接 |
| 流式写入 | 支持 | 支持 | 不支持 |
| 日志轮转 | `pino-roll` 扩展 | 内置 `File` transport | 无 |
| Express 集成 | `pino-http`（成熟） | `express-winston` | 无 |
| Alpine 兼容 | ✅ 零依赖 | ✅ | ✅ |
| ESM 支持 | ✅ v9+ | ⚠️ 部分 | ✅ |
| 包体积 | ~1.2MB | ~4.5MB | 0 |
| Docker 友好 | stdout JSON 一行一对象 | 需额外配置 | 可用 |

### 1.2 选定：pino

**理由**：

1. **性能** — Node.js 生态中最快的日志库。每个日志条目仅增加微秒级开销（`benchmark: ~750k ops/sec vs winston ~140k`），对高频率答题 API 调用尤为重要。
2. **JSON 原生** — 每行一个 JSON 对象，天然适合 Docker 的 `json-file` 日志驱动和后续对接 `journald`/`loki` 等日志系统。
3. **低体积** — Alpine 容器中节省空间。
4. **Express 生态** — `pino-http` 自动记录每个 HTTP 请求的 `method` `url` `status` `responseTime`，无需手动打点。
5. **轮转扩展** — `pino-roll` 提供基于大小和时间的文件轮转。

### 1.3 依赖包

```json
{
  "dependencies": {
    "pino": "^9.6.0",
    "pino-http": "^10.4.0",
    "pino-roll": "^1.1.0"
  }
}
```

三者在 `node:20-alpine` 上无原生编译依赖，安装开销极小。

## 2. 日志级别

遵循 pino 的 RFC 5424 级别体系（数字越小越严重）：

| 级别 | 值 | 使用场景 |
|------|----|---------|
| `fatal` | 60 | 进程无法继续运行的致命错误（如数据库连接失败、端口被占用） |
| `error` | 50 | 业务错误或异常（如登录失败、参数校验不通过、数据库查询异常） |
| `warn` | 40 | 值得关注的异常情况（如 API 调用频率过高、用户尝试越权操作） |
| `info` | 30 | **默认级别**。正常业务流程记录（如用户登录、答题提交、场次完成） |
| `debug` | 20 | 开发调试信息（如 SQL 查询、请求参数、SM-2 中间计算结果） |
| `trace` | 10 | 全量追踪（如函数调用栈、数据库连接池状态），仅本地开发时开启 |

**生产环境默认级别**：`info`（`debug` 和 `trace` 不输出，减少磁盘写入）。

## 3. 日志分类与内容

### 3.1 HTTP 请求日志（pino-http 自动）

**触发**：每个进入 Express 的 HTTP 请求

| 字段 | 说明 | 示例 |
|------|------|------|
| `req.method` | HTTP 方法 | `POST` |
| `req.url` | 请求路径 | `/api/learn/answer` |
| `req.headers` | 请求头（脱敏） | 隐藏 `authorization` 完整 token |
| `res.statusCode` | 响应状态码 | `200` |
| `responseTime` | 处理时长（ms） | `45` |
| `req.id` | 请求 ID（关联链路） | `req-1` |

**示例输出**：

```json
{"level":30,"time":1716278400000,"pid":1,"req":{"method":"POST","url":"/api/learn/answer"},"res":{"statusCode":200},"responseTime":45,"msg":"request completed"}
```

> `pino-http` 自动脱敏 `req.headers.authorization`，只保留 `Bearer [redacted]`。

### 3.2 应用业务日志

#### 3.2.1 认证日志

| 事件 | 级别 | 内容 |
|------|------|------|
| 用户注册成功 | `info` | `userId`, `username` |
| 用户注册失败（用户名已存在） | `warn` | `username`, `reason` |
| 用户登录成功 | `info` | `userId`, `username`, `ip` |
| 用户登录失败（密码错误） | `warn` | `username`, `ip`, `reason` |
| 用户登录失败（账号冻结） | `warn` | `userId`, `reason` |
| Token 验证失败 | `warn` | `reason`, `ip` |

#### 3.2.2 学习流程日志

| 事件 | 级别 | 内容 |
|------|------|------|
| 创建场次 | `info` | `userId`, `sessionId`, `round`, `mode`（new/review/mixed） |
| 提交答题 | `info` | `userId`, `wordId`, `sessionId`, `correct`, `responseTimeMs` |
| 场次完成 | `info` | `userId`, `sessionId`, `wordsReviewed`, `wordsPassed`, `duration` |
| 轮次完成 | `info` | `userId`, `round`, `wordsMastered`, `totalWords`, `avgProficiency` |

#### 3.2.3 管理员操作日志

| 事件 | 级别 | 内容 |
|------|------|------|
| 重置用户密码 | `info` | `adminUserId`, `targetUserId` |
| 设置/取消管理员 | `info` | `adminUserId`, `targetUserId`, `newAdminStatus` |
| 冻结/解冻用户 | `info` | `adminUserId`, `targetUserId`, `newFrozenStatus` |
| 删除用户 | `warn` | `adminUserId`, `targetUserId`, `deletedReviewLogs`, `deletedSessions` |

> 管理员操作类日志等级可设为 `info` 或 `warn`，建议 **`info`**。删除用户操作提升为 `warn` 以突出敏感度。

#### 3.2.4 系统与错误日志

| 事件 | 级别 | 内容 |
|------|------|------|
| 服务启动 | `info` | `port`, `nodeVersion`, `dbPath` |
| 数据库连接失败 | `fatal` | `error.message`, `error.stack` |
| 未捕获异常 | `fatal` | `error.message`, `error.stack` |
| Promise rejection | `error` | `reason`, `promise` |
| 数据库慢查询（>500ms） | `warn` | `sql`, `duration` |
| 种子脚本执行 | `info` | `wordsInserted`, `totalWords` |

## 4. 输出格式（JSON Schema）

所有日志使用 pino 的默认 JSON 格式，每行一个对象（NDJSON）。

### 4.1 基础字段

```typescript
interface LogEntry {
  /** 日志级别数字（10=TRACE, 20=DEBUG, 30=INFO, 40=WARN, 50=ERROR, 60=FATAL） */
  level: number;
  /** Unix 毫秒时间戳 */
  time: number;
  /** 进程 PID */
  pid: number;
  /** 主机名（容器内为容器 ID 前 12 位） */
  hostname: string;
  /** 日志消息 */
  msg: string;

  // ─── 以下为可选扩展字段 ───

  /** 请求链路 ID（自动生成，用于关联同一请求的多个日志） */
  reqId?: string;
  /** 当前登录用户 ID（已认证的请求） */
  userId?: number;
  /** 当前场次 ID（学习流程中） */
  sessionId?: number;
  /** 当前轮次 */
  round?: number;
  /** 错误堆栈（仅在 error/fatal 级别出现） */
  err?: {
    type: string;
    message: string;
    stack: string;
  };
}
```

### 4.2 HTTP 请求日志特有字段（pino-http）

```typescript
interface HttpLogEntry extends LogEntry {
  req: {
    method: string;        // "GET" | "POST" | "PUT" | "DELETE"
    url: string;           // "/api/learn/answer"
    headers: object;       // 含 content-type, user-agent 等，authorization 已脱敏
    remoteAddress: string;
  };
  res: {
    statusCode: number;     // 200, 401, 500 等
  };
  responseTime: number;     // 单位 ms
}
```

### 4.3 日志示例

**普通业务日志**（info）：

```json
{"level":30,"time":1716278400000,"pid":1,"hostname":"2cba81a5bf0c","reqId":"req-abc123","userId":42,"sessionId":15,"msg":"用户提交答题 - correct=true, wordId=1234, responseTimeMs=3200"}
```

**警告日志**（warn）：

```json
{"level":40,"time":1716278401000,"pid":1,"hostname":"2cba81a5bf0c","userId":7,"msg":"登录失败 - 密码错误","username":"testuser","ip":"192.168.1.100"}
```

**错误日志**（error）：

```json
{"level":50,"time":1716278402000,"pid":1,"hostname":"2cba81a5bf0c","reqId":"req-def456","msg":"数据库查询异常","err":{"type":"Error","message":"SQLITE_ERROR: no such table: words","stack":"Error: ..."}}
```

## 5. 存储策略

### 5.1 日志文件路径

```
/app/logs/                          # 容器内路径，volume 挂载到宿主机 ../logs/
├── app-2026-05-21.log              # 主日志（info 及以上）
├── app-2026-05-22.log              # 每日轮转
├── error-2026-05-21.log            # 错误日志（warn 及以上，冗余存储便于快速定位）
└── error-2026-05-22.log
```

| 路径 | 宿主机映射 | 权限 |
|------|-----------|------|
| `/app/logs/` | `../logs/`（`docker-compose.yml` volume 定义） | `1001:1001`（`start.sh` 所有权修复） |

### 5.2 双重输出

日志同时写入两个目标：

| 目标 | 用途 | 实现方式 |
|------|------|----------|
| **stdout**（容器标准输出） | Docker 日志驱动（`docker logs`），生产实时查看 | pino 默认写入 stdout |
| **文件**（`/app/logs/`） | 持久化存储，历史回溯 | pino 多 stream（`pino.multistream`） |

```
         ┌──────────────────┐
         │    Application   │
         └────────┬─────────┘
                  │ logger.info(...)
                  │
         ┌────────┴────────┐
         │  pino instance  │
         └────────┬────────┘
                  │
         ┌────────┴────────┐
         │ multistream     │
         └──┬──────────┬───┘
            │          │
     stdout │          │ file transport
            ▼          ▼
   ┌────────────┐  ┌──────────────────┐
   │ docker logs│  │ /app/logs/       │
   │ (容器 stdout)│  │ app-2026-05-21.log│
   └────────────┘  └──────────────────┘
```

### 5.3 文件轮转

使用 `pino-roll` 按**时间**轮转：

| 参数 | 值 | 说明 |
|------|----|------|
| 轮转周期 | `daily` | 每天 00:00 创建新文件 |
| 文件名模式 | `app-%Y-%m-%d.log` | 日期后缀 |
| 错误日志文件名 | `error-%Y-%m-%d.log` | 独立错误日志 |
| 压缩 | 否 | Alpine 容器节省 CPU，不启用 gzip |
| 同步写入 | 是（`sync: true`） | 防止容器崩溃时丢失最后几条日志 |

## 6. 过期与清理

### 6.1 保留策略

| 日志类型 | 保留天数 | 说明 |
|----------|---------|------|
| 主日志（app-*.log） | **30 天** | 覆盖一个完整学习轮次周期 |
| 错误日志（error-*.log） | **60 天** | 错误日志量小，保留更长时间利于回溯 |
| stdout（Docker 日志） | 由 Docker 日志驱动决定 | 建议 `docker compose` 中配置 `logging: max-size: "10m" max-file: "3"` |

### 6.2 清理机制

**方案一（首选）：应用启动时自动清理**

在 `src/server/index.ts` 中，服务启动时执行清理：

```typescript
import fs from 'fs'
import path from 'path'

const LOG_DIR = '/app/logs'
const APP_LOG_RETENTION_DAYS = 30
const ERROR_LOG_RETENTION_DAYS = 60

function cleanOldLogs(): void {
  const now = Date.now()
  const files = fs.readdirSync(LOG_DIR)
  for (const file of files) {
    const filePath = path.join(LOG_DIR, file)
    const stat = fs.statSync(filePath)
    const ageDays = (now - stat.mtimeMs) / (1000 * 60 * 60 * 24)

    if (file.startsWith('error-') && ageDays > ERROR_LOG_RETENTION_DAYS) {
      fs.unlinkSync(filePath)
      console.log(`清理过期错误日志: ${file}`)
    } else if (file.startsWith('app-') && ageDays > APP_LOG_RETENTION_DAYS) {
      fs.unlinkSync(filePath)
      console.log(`清理过期主日志: ${file}`)
    }
  }
}
```

- **触发时机**：Express 服务 `listen()` 之前
- **优点**：零额外依赖，每次重启自动清理
- **容错**：日志文件被其他进程占用时跳过，不阻塞启动

**方案二（辅助）：pino-roll 内置 maxFiles**

```typescript
// 配置 pino-roll 的 maxFiles（保留最近 N 个文件）
// 例：保留最近 31 个日切文件（1 个当前 + 30 个历史）
{
  frequency: 'daily',
  maxFiles: 31,
}
```

> 两种方案可同时启用，互为兜底。

### 6.3 磁盘占用预估

| 项目 | 估算 | 说明 |
|------|------|------|
| 每日主日志 | ~5-10 MB | 假设 2000 次 API 请求 + 业务日志 |
| 每日错误日志 | ~0.1-0.5 MB | 错误量远少于正常请求 |
| **30 天总量** | **~150-300 MB** | 完全可接受 |
| **60 天错误日志** | **~6-30 MB** | 可忽略 |

> 实际数据取决于用户活跃度。`logs/` volume 所在磁盘应有至少 1GB 空闲空间。

## 7. Docker 集成

### 7.1 docker-compose.yml 日志配置

```yaml
services:
  cet4-web:
    # ... 其他配置不变 ...
    logging:
      driver: "json-file"
      options:
        max-size: "10m"       # 每个日志文件最大 10MB
        max-file: "3"         # 保留最近 3 个文件
```

这是对 Docker 引擎层日志的约束（`docker logs` 输出），与应用层文件日志独立互补。

### 7.2 Dockerfile 变更

无需变更。pino 为纯 JS 包，零原生编译依赖，`npm ci` 自动安装。

### 7.3 日志目录所有权

`logs/` 目录已由 `start.sh` 统一管理：

```bash
# start.sh 自动执行
sudo chown -R "$CURRENT_UID:$CURRENT_GID" ./data ./config ./logs
```

## 8. 代码接入方案

### 8.1 logger 模块结构

```
src/server/lib/
└── logger.ts              # pino 实例创建与配置
```

### 8.2 logger.ts 核心代码

```typescript
import pino from 'pino'
import pinoHttp from 'pino-http'
import fs from 'fs'
import path from 'path'

const LOG_DIR = '/app/logs'
const APP_LOG_RETENTION_DAYS = 30
const ERROR_LOG_RETENTION_DAYS = 60
const LOG_LEVEL = process.env.LOG_LEVEL || 'info'

// 确保日志目录存在
fs.mkdirSync(LOG_DIR, { recursive: true })

// 启动时清理过期日志（见 6.2 节）
cleanOldLogs()

// 主日志文件 transport
const appTransport = pino.transport({
  target: 'pino-roll',
  options: {
    file: path.join(LOG_DIR, 'app-%Y-%m-%d.log'),
    frequency: 'daily',
    maxFiles: 31,     // 保留 31 天
    mkdir: true,
    symlink: false,
  },
})

// 错误日志文件 transport（仅 warn 及以上）
const errorTransport = pino.transport({
  target: 'pino-roll',
  options: {
    file: path.join(LOG_DIR, 'error-%Y-%m-%d.log'),
    frequency: 'daily',
    maxFiles: 61,     // 保留 61 天
    mkdir: true,
  },
  level: 'warn',      // 只接收 warn 及以上级别
})

// 创建 pino 实例（多 stream + stdout）
const logger = pino(
  {
    level: LOG_LEVEL,
    // 生产环境禁用 pino 的漂亮打印（全 JSON）
    ...(process.env.NODE_ENV === 'development' && {
      transport: {
        target: 'pino-pretty',
        options: { colorize: true },
      },
    }),
  },
  pino.multistream([
    { stream: appTransport },           // → 文件 app-*.log
    { stream: errorTransport },         // → 文件 error-*.log
    { stream: process.stdout },         // → Docker stdout
  ])
)

// pino-http 中间件工厂
const httpLogger = pinoHttp({
  logger,
  // 脱敏 authorization 头
  redact: ['req.headers.authorization'],
  // 自定义请求日志级别：4xx → warn，5xx → error
  customLogLevel: (req, res, err) => {
    if (res.statusCode >= 500 || err) return 'error' as const
    if (res.statusCode >= 400) return 'warn' as const
    return 'info' as const
  },
})

export { logger, httpLogger }
```

### 8.3 在 Express 中接入

```typescript
// src/server/index.ts
import express from 'express'
import { logger, httpLogger } from './lib/logger.js'

const app = express()

// 必须在所有路由之前注册
app.use(httpLogger)

app.get('/api/health', (req, res) => {
  res.json({ status: 'ok' })
})

// 业务代码中使用
app.post('/api/learn/answer', (req, res) => {
  logger.info({ userId: req.user.id, wordId: req.body.wordId, correct: req.body.correct }, '提交答题')
  // ...
})

// 未捕获异常
process.on('uncaughtException', (err) => {
  logger.fatal({ err }, '未捕获异常，进程退出')
  process.exit(1)
})

process.on('unhandledRejection', (reason) => {
  logger.error({ err: reason }, '未处理的 Promise rejection')
})

const PORT = process.env.PORT || 9098
app.listen(PORT, () => {
  logger.info({ port: PORT, nodeVersion: process.version }, '服务启动')
})
```

### 8.4 在 CLI 脚本中使用

CLI 脚本（如 `reset-password.js`）由于运行在终端交互环境，使用独立的控制台输出：

```typescript
// cli/reset-password.js — 保持 console.log 输出
// 但可通过环境变量登录到文件：
// RESET_PASSWORD_LOG=/app/logs/reset-password.log
if (process.env.RESET_PASSWORD_LOG) {
  fs.appendFileSync(process.env.RESET_PASSWORD_LOG,
    `[${new Date().toISOString()}] reset-password: ${username} → ${newPassword}\n`)
}
```

> CLI 脚本不引入 pino，避免增加启动复杂度。

## 9. 附录

### 9.1 与 review_logs 数据库表的关系

| 维度 | 应用日志（本设计） | review_logs 表 |
|------|-------------------|----------------|
| 用途 | 系统运行跟踪、问题排查 | 学习数据持久化、SM-2 算法输入 |
| 数据源 | pino logger | application code → database.ts |
| 存储 | 文件（`/app/logs/`） | SQLite（`data/cet4.db`） |
| 保留期 | 30~60 天 | 永久（用户生命周期内） |
| 是否可恢复 | 否（文件删除即丢失） | 是（数据库备份可恢复） |
| 查询方式 | grep / docker logs / 文件浏览器 | SQL 查询 + API |
| 脱敏要求 | 需要（隐藏密码、token） | 不需要（仅存 user_id） |

两者为**互补关系**：应用日志用于运维排查，`review_logs` 用于学习算法和进度展示。

### 9.2 敏感数据脱敏规则

| 数据类型 | 日志输出策略 |
|----------|-------------|
| 密码原文 | **绝不记录** |
| JWT token | `req.headers.authorization` → `Bearer [redacted]`（pino-http 自动） |
| 用户邮箱 | 记录完整邮箱（非敏感信息） |
| 答题正误 | 记录（`correct: true/false`） |
| 用户密码 hash | **绝不记录** |

### 9.3 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `LOG_LEVEL` | `info` | 日志级别：`trace` `debug` `info` `warn` `error` `fatal` |
| `NODE_ENV` | `production` | `development` 时启用 `pino-pretty` 彩色控制台输出 |
| `LOG_DIR` | `/app/logs` | 日志文件输出目录（一般不改） |
| `LOG_RETENTION_DAYS` | `30` | 主日志保留天数 |
